# Архитектура приложения «ЦЕХ»

Система управления швейным цехом. Frontend — vanilla JS SPA (без сборки), backend — Supabase PostgreSQL. Вся бизнес-логика и контроль доступа реализованы в **RPC-функциях PostgreSQL** (`SECURITY DEFINER`). Клиент не имеет прямого доступа к таблицам — только через функции, которые проверяют сессию и роль. Это гарантирует целостность расчётов и защиту от обхода прав.

---

## 1. Таблицы и связи

| Таблица | Назначение | Ключевые связи |
|---|---|---|
| `employees` | Сотрудники, роли, PIN-хэш, статус | — |
| `sessions` | Токены сессий (auth) | `employee_id → employees` |
| `batches` | Производственные партии | `created_by → employees` |
| `tasks` | Задачи по партии и сотруднику | `batch_id → batches`, `employee_id → employees` |
| `work_records` | **Источник правды для зарплаты** | `task_id → tasks`, `batch_id`, `employee_id` |
| `payments` | Выплаты (salary/advance/bonus/penalty) | `employee_id`, `batch_id` |
| `advance_requests` | Запросы аванса | `employee_id`, `resolved_by` |
| `expenses` | Расходы цеха / партии | `batch_id` (nullable) |
| `notifications` | Уведомления внутри приложения | `recipient_id` или `recipient_role` |

Все `id` — UUID (`gen_random_uuid()`). Везде есть `created_at`; где уместно — `updated_at` (поддерживается триггером).

## 3. Основные пользовательские сценарии

1. **Вход:** телефон + PIN → функция `login` сверяет `pin_hash` (bcrypt), создаёт сессию, возвращает токен.
2. **Создание партии (admin):** партия создаётся со `status=active`, `current_stage=cutting`. Уведомление технологу.
3. **Назначение задач (admin/technologist):** на каждом этапе создаются задачи со ставкой `rate_per_unit`. Уведомление исполнителю.
4. **Выполнение работы (исполнитель):** «Отметить выполнение» → вводит количество → `submit_work` создаёт `work_record`, обновляет `completed_quantity` и статус задачи.
5. **Перевод этапа (admin/technologist):** при незавершённых задачах — предупреждение; admin может перевести принудительно. После `packing` партия → `completed`.
6. **Зарплата (admin):** видит начислено/выплачено/остаток, делает выплату/аванс/бонус/штраф.
7. **Запрос аванса (любой):** создаёт запрос → уведомление admin → admin approve/reject/paid (paid создаёт payment).
8. **Расходы (admin):** добавляет расход с привязкой к партии или общий.
9. **Себестоимость:** карточка партии считает ткань + расходы + работа, прибыль и маржа.

## 4. Правила расчёта зарплаты

Зарплата считается **только из `work_records`** (никогда напрямую из `tasks`).

```
total_amount (на запись) = quantity * rate_per_unit

Начислено = Σ work_records.total_amount  (по employee_id)
Выплачено = Σ payments.amount  где type ∈ {salary, advance}
Бонусы    = Σ payments.amount  где type = bonus
Штрафы    = Σ payments.amount  где type = penalty
Остаток   = Начислено + Бонусы − Штрафы − Выплачено
```

Задача `stage = technologist` участвует в расчёте так же, как любая другая (ставка × готовые единицы).

## 5. Правила перехода партии по этапам

Конвейер: `cutting → sewing → ironing → packing → completed`.

- Партия создаётся: `active`, `cutting`.
- Перевод доступен admin и technologist.
- Перед переходом проверяется: все **не отменённые** задачи текущего этапа со статусом `completed`. Если нет — функция возвращает признак `needs_force`; UI показывает предупреждение.
- Admin может перевести принудительно (`force=true`) — UI требует подтверждения.
- После `packing` → `current_stage=completed`, `status=completed`, `completed_at=now()`.
- Откат завершённой партии — только admin и только с подтверждением.
- Каждый переход создаёт уведомление.

---

## 6. Защита от двойного начисления

1. `submit_work` блокирует строку задачи (`SELECT ... FOR UPDATE`) — конкурентные вызовы сериализуются.
2. Проверка `completed_quantity + quantity <= planned_quantity` — нельзя превысить план.
3. Проверка `quantity > 0`.
4. Нельзя создать `work_record` без `task_id`, по отменённой задаче, или чужой задаче.
5. На клиенте кнопка блокируется на время запроса (антидабл-клик).
6. Каждая частичная сдача — отдельный `work_record` (накопительно обновляет `completed_quantity`).

---

## 7. Безопасность авторизации

- PIN хранится только как `pin_hash` (bcrypt через `pgcrypto.crypt`).
- 5 неверных попыток → блокировка входа на 5 минут (`locked_until`).
- `dismissed` не может войти.
- Сессия — токен в таблице `sessions` (срок 30 дней), хранится в `localStorage`; при выходе удаляется и на сервере, и локально.
- Таблицы недоступны клиенту напрямую (RLS deny-all для anon/authenticated); всё идёт через `SECURITY DEFINER` функции, проверяющие токен.
